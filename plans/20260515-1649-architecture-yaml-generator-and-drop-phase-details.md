# Architecture YAML generator + drop diagram-phase-details

Status: draft (open questions OQ1–OQ4 unresolved)
Author: claude (conversation 2026-05-15)

## Background

Two ATDD diagram files live under `internal/assets/global/docs/atdd/` and get scaffolded into every student repo:

- `internal/assets/global/docs/atdd/architecture/diagram-architecture.md` — overview + 7 sub-diagrams (Test, DSL Port, DSL Core, Driver Port, Driver Adapter Shop, Driver Adapter External, System) describing the student repo's ATDD layered architecture.
- `internal/assets/global/docs/atdd/process/diagram-phase-details.md` — Mermaid flowcharts of per-phase WRITE/COMPILE/RUN/COMMIT mechanics for AT-RED-TEST, AT-GREEN-SYSTEM, CT-RED-TEST, etc.

Both files declare themselves auto-generated by a `diagram-generator` agent. That agent no longer exists in `.claude/agents/`. Five meta-agents still reference it as a live dependency: `diagram-content-editor.md`, `diagram-tweaker.md`, `token-usage-audit.md`, `process-audit.md`, `architecture-sync.md`. They also reference a third file `diagram-process.md` that has never existed on disk.

Separately, gh-optivem already has a YAML-driven code generator for its own runtime orchestration: `internal/atdd/runtime/statemachine/process-flow.yaml` → `internal/atdd/runtime/diagram/diagram.go` → `docs/process-diagram.md`, regenerated by `.github/workflows/regenerate-diagram.yml`.

## Goal

1. Replace the missing `diagram-generator` agent with a code generator for the architecture diagram, mirroring the `process-flow.yaml` pattern.
2. Drop `diagram-phase-details.md` entirely. Per-phase mechanics already live in the prose docs (`at-red-test.md`, `at-green-system.md`, `ct-red-test.md`, …). A third copy in a generated diagram only creates drift between prose and diagram.
3. Sweep the five meta-agents so no dangling `diagram-generator` / `diagram-process` / `diagram-phase-details` references remain.

## Non-goals

- The phase-details prose docs themselves — unchanged. They are now the *only* source of truth for per-phase mechanics.
- gh-optivem's own `process-flow.yaml` / `docs/process-diagram.md` pipeline — unchanged.
- Re-creating a `diagram-process.md` cycle-level diagram for the student repos. Cycle-level orchestration is already in gh-optivem's `docs/process-diagram.md`; students reading their scaffolded repo can follow the link rather than carry a regenerated copy.
- Reviving any other removed agent. This plan only addresses the dangling diagram-generator references.

## Open questions (resolve before coding)

### OQ1. Output path for the rendered architecture diagram

The current file lives at `internal/assets/global/docs/atdd/architecture/diagram-architecture.md` (scaffold tree — copied into each student repo). The conversation that produced this plan used the phrase `docs/architecture-diagram.md`, which would put it in gh-optivem's own `docs/`. But the diagram describes the **student repo's** ATDD architecture (DSL Port stages, Driver Adapter shop/external split, System backends), not gh-optivem's own runtime.

Options:

- **(A) Keep in scaffold tree only.** Output stays at `internal/assets/global/docs/atdd/architecture/diagram-architecture.md`. Students see it in their repo; gh-optivem's own docs do not duplicate it.
- **(B) Move to gh-optivem docs.** Output at `docs/architecture-diagram.md`. Remove from scaffold tree. Students follow a link to gh-optivem to see the diagram (mirrors how cycle-level orchestration is handled today).
- **(C) Both.** Render once into gh-optivem's `docs/`, then copy/symlink/regenerate into the scaffold tree. Two writes per regeneration.

Recommendation: **(A)** — the diagram is student-facing content describing the student's own repo; keeping it in the scaffold tree matches what's there today and avoids the cross-repo link.

### OQ2. Where does `architecture.yaml` live?

Options:

- **(A) `internal/atdd/runtime/architecture/architecture.yaml`** — mirrors `internal/atdd/runtime/statemachine/process-flow.yaml`. gh-optivem owns it as canonical input; the renderer reads it via `go:embed` like the statemachine does.
- **(B) `internal/assets/global/docs/atdd/architecture/architecture.yaml`** — lives next to its output. But then it's also scaffolded into student repos as a YAML file with no consumer there, which is confusing.
- **(C) `internal/atdd/runtime/diagram/architecture.yaml`** — co-located with the renderer.

Recommendation: **(A)**, for symmetry with `process-flow.yaml`.

### OQ3. YAML schema

The current file has 7 sub-diagrams that share a common shape: heading + node list + edge list, with three edge styles:

- directional: `A --> B`
- undirected: `A --- B`
- labeled directional: `A -->|label| B`

Some nodes are cross-section references (`DSL Port — see § DSL Port`). Some labels contain arbitrary text (`withFieldName method`, `ShopApiClient`).

Proposed schema:

```yaml
sections:
  - name: overview              # heading "Overview" / "Test" / "DSL Port" / ...
    direction: TD               # mermaid flowchart direction (default TD)
    nodes:
      - id: TEST
        label: Test Files
      - id: DSL_PORT
        label: DSL Port
        ref: dsl_port           # marks as cross-section ref; renders as "DSL Port — see § DSL Port"
    edges:
      - {from: TEST, to: DSL_PORT}                            # directional
      - {from: GIVEN_STAGE, to: WITH_STRING, style: undirected}
      - {from: GIVEN_STAGE, to: WHEN_STAGE, label: when}      # labeled directional
```

Order of `sections:`, `nodes:`, `edges:` is the render order — deterministic.

Recommendation: above schema. Keep it boring; no inheritance, no node-typing (architecture diagrams don't need BPMN-style executor coloring), no groups.

### OQ4. CLI surface

Existing pattern: `gh optivem process show > docs/process-diagram.md` (defined in `process_commands.go`).

Options:

- **(A) `gh optivem architecture show`** — new top-level command, parallels `process show`.
- **(B) `gh optivem diagram show <name>`** — generalized; would also subsume the existing process diagram.

Recommendation: **(A)** for now — symmetry with the existing command, no breaking change to `process show`. A future generalization is cheap if we add a third diagram.

## Risks

- **Shop-specific labels in `global/`.** The current `diagram-architecture.md` carries shop-specific node labels (`OrderController`, `CouponController`, `ProductController`, `system/monolith/java`, …) inside `internal/assets/global/` which is supposed to be template-agnostic. This is a pre-existing issue. Translating verbatim into YAML preserves it — call it out, do not silently rename. Cleanup is a follow-up plan.
- **Five meta-agents reference the dead generator.** Between deciding "keep the architecture diagram" and landing step 8, any of those agents may instruct a user to "run `diagram-generator` first." Mitigation: do steps 5/6/8 in one PR so the new generator and the agent sweep land together.
- **Golden test byte-stability.** Mermaid map iteration order in Go can shuffle output. The current `diagram.go` uses sorted slices for determinism — the new renderer must do the same.
