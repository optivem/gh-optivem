# Ticket `subtype:*` label values — verb → noun

🤖 **Picked up by agent** — `Valentina_Desk` at `2026-05-26T10:23:36Z`

**Date:** 2026-05-26
**Origin:** Conversation 2026-05-26. The `ticket-kind` gate's error message (commit `f3df7bd`) now enumerates valid `subtype:*` label values: `cover-legacy`, `redesign-system`, `refactor-system`, `refactor-tests`, `onboard-external-system`. All five are verb-form, but the parent ticket-type set (`story`, `bug`, `task`) is noun-form — so the subtype reads as "this task is [do thing]" instead of the parallel "this task is a [thing]". The verb form also collides grammatically with PR titles and commit messages ("Refactor system" the action vs. the label).

## Decision

Rename label values from verb form to noun form. The `subtype:*` label values change; **cycle / process / phase names stay as verbs** (they describe what the cycle does, so verb form is correct and parallel with `change-system-behavior`).

| current value (verb) | new value (noun) |
|---|---|
| `cover-legacy` | `legacy-coverage` |
| `redesign-system` | `system-redesign` |
| `refactor-system` | `system-refactor` |
| `refactor-tests` | `test-refactor` |
| `onboard-external-system` | `external-system-onboarding` |

`ticket-kind` discriminator strings (consumed by `process-flow.yaml` edge guards) change accordingly: `task/cover-legacy` → `task/legacy-coverage`, etc.

## Open questions (resolved upfront, per [[feedback_resolve_questions_upfront]])

- **Q1 — Hyphenation form.** Use `<adjective>-<noun>` head order: `legacy-coverage`, `system-redesign`, `external-system-onboarding`. Last word is the head noun. Confirmed in table above.

- **Q2 — Retain `task/` prefix.** Yes. The `ticket-kind` gate emits values like `task/<subtype>` to discriminate against `story` / `bug` in the same downstream switch. Only the value after the slash changes.

- **Q3 — Existing GitHub-issue labels.** Operators must relabel any in-flight ticket whose current `subtype:*` label is in the verb form. Manual one-time step; flag in Item 6's verification and in the merge PR description. There is no migration script — `gh-optivem` is teaching-repo, not legacy-tolerant (per [[feedback_teaching_repo_no_legacy]]).

- **Q4 — Bundling.** Items 1–4 are individually-committable rename steps with passing tests at each step. Item 5 is a separate diagram-regen commit. Item 6 is verification, not a commit. Six commits total expected — small enough that bundling into one is acceptable if reviewer prefers, but the per-file split keeps the history readable.
