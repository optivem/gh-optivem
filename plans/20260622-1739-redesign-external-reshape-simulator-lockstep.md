### Q6 (RESOLVED 2026-06-23 → Option A): how does the redesign cycle acquire per-system identity + real-kind?

The CT leg the plan asked to "reuse verbatim" depends on machinery the redesign cycle does not have. `UnrollExternalSystems` (channels.go:199) rewrites exactly ONE anchor — `IMPLEMENT_AND_VERIFY_EXTERNAL_DRIVER_ADAPTERS` in shared-contract — into per-system clones that bake `external-system-name` + `real-kind` into params and stamp `real-kind` into state (via `resolve-external-system`). `redesign-external-system-structure` is reached from the implement-ticket `external-system-redesign` subtype gate and the refactor menu — **neither unrolls it**.

Options considered:

- **(A) Unroll the redesign cycle per external system** — add `redesign-external-system-structure` as a second `UnrollExternalSystems` anchor (clone its per-system body per registered system, baking `external-system-name` + `real-kind`, with a `resolve-external-system` + `GATE_EXTERNAL_SYSTEM_TOUCHED` self-guard like the CT cycle).
- **(B) Real-kind-free, single-system composition** — drop `GATE_CONTRACT_REAL_RED_KIND` + the upstream-gap halt; after the adapter reshape, probe contract-real and on RED go straight to the simulator implementer. Sheds the real-kind safety branch the plan explicitly wanted, and **still needs** `external-system-name` resolution + a contract-test selection — so it does not escape the gap.
- **(C) Resolve identity without unrolling** — add a redesign-specific `resolve-*` service-task at the cycle start that derives the target system(s) + real-kind from the redesign ticket and stamps them into state, then run the leg whole-suite.

**RESOLUTION: (A).** The original draft recommended (C); it is overturned on one factual correction plus a fair re-weighting:

1. **The disqualifying sub-problem for (A) is obsolete.** The draft rejected (A) because "the per-system touched-guard reads `external-driver-port-changed-paths`, which the redesign path never populates." But `touchedExternalSystemNames` (external.go:59) now resolves by **ESCC-first precedence** (plan 20260620-1850, landed after this Q6 was written): `escc-systems` is authoritative when `ticket-has-escc`, and `parse-ticket` stamps both at intake for *every* ticket. A contract-reshape ticket (#81 reshapes the ERP `GetProductResponse` contract) carries ESCC by nature. The selection source exists; (A)'s blocker does not. The only new code is a loud guard requiring ESCC on the redesign-external entry (Step 4c) — without it, an ESCC-less redesign ticket silently no-ops (the #65-class bug).
2. **(A) follows the architecture's grain; (C) fights it.** external.go:131 records that runtime identity resolution (`identify-external-system`) was **deliberately retired** in favour of load-time baking via `UnrollExternalSystems`. (C) re-introduces exactly that retired pattern as a parallel resolver — a second source of truth for system-name + real-kind that can drift from the CT cycle's. (A) reuses `unrollAnchor` (already generic, channels.go:246) and `resolve-external-system` verbatim.
3. **The "less machinery" advantage of (C) is smaller than it looked.** Both (A) and (C) must reuse the reconcile leg, which is embedded mid-CT-HIGH — so **both require the Step 4a extract**. (C) saves the second unroll anchor but pays it back in a bespoke resolver + whole-suite probe selection.

Honest cost of (A) we accept: higher blast radius (a second load-time anchor touches graph rewriting the CT cycle depends on) and the statemachine-test loopback/RAM hazard (`feedback_statemachine_test_loop_hazard`) — mitigated by Step 4a being behavior-preserving (existing CT tests gate it) and by auditing gate fixtures before running statemachine tests. (B) is rejected outright: it sheds the wanted real-kind safety branch and saves almost nothing (real-kind is free from the registry once the name is known).
