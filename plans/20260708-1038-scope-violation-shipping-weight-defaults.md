## TL;DR

**Why:** Rehearsal #72 (Charge shipping based on product weight from ERP, machine `ValentinaLaptop`, run `20260708-083815`) halted at `implement-system` (channel=api) after 21m30s with `STOP_SCOPE_VIOLATION`. `system-implementer` correctly refused (per its own documented "contradictory tests" instructions) because a new default value (`ScenarioDefaults.DEFAULT_WEIGHT = "1.00"`, added by `dsl-implementer` at `ScenarioDefaults.java:14` / wired at `GivenProductImpl.java:21`) silently changed the computed total for a pre-existing test (`PlaceOrderPositiveTest.totalPriceShouldBeSubtotalPricePlusTaxAmount`) once shipping became a new formula term — and the default wasn't even needed, since the ticket's own new AC scenarios set weight explicitly. `acceptance-test-writer`'s existing regression-reconciliation heuristic (literal grep of the new rule's vocabulary) never fired because the broken old test doesn't mention "shipping" at all. On top of that, the generic `STOP_SCOPE_VIOLATION` halt message told the human to "widen scope" — wrong advice for a contradictory-tests case, whose fix is reconciling a test, not widening `system-implementer`'s write scope.
**End result:** `dsl-implementer` no longer invents load-bearing nonzero defaults the ticket's own tests don't need; a contradictory-tests scope-exception gets its own correctly-worded halt instead of the generic "widen scope" message; `acceptance-test-writer`'s regression heuristic also catches silent default-driven ripple effects, not just literal-vocabulary matches; and a new early verify step catches DSL/driver-layer regressions (a related but distinct failure class) right after DSL/driver changes land.

## Outcomes

- `dsl-implementer` picks defaults for newly-formula-relevant fields that don't retroactively perturb pre-existing computed assertions when the ticket's own new scenarios already supply the real value explicitly.
- A prod-agent scope-exception that names acceptance-test files (a contradictory-tests case) routes to its own actionable halt event telling the human to reconcile the test via `acceptance-test-writer` — never the generic "widen scope" `STOP_SCOPE_VIOLATION` message.
- `acceptance-test-writer`'s "Behavior-change tickets" reconciliation trigger also catches pre-existing tests broken by a newly-added/changed shared default (not just tests using the new rule's own vocabulary).
- The BPMN runs the pre-existing (non-new) test suite immediately after a DSL/driver change lands, catching DSL-layer regressions early — documented as a distinct, additional safety net, not a fix for the #72 root cause itself (which only becomes observable once production code is written).
