# 2026-06-15 18:45:00 UTC — Fix-flow stall: interactive human-dispatch in unattended runs + fixer frame-mismatch

## TL;DR

**Why:** Run #69 (`Reject order with line quantity of 100`) burned **2h47m wall-clock and then halted on a bogus infra error**, and almost all of that time (2h14m58s) was a single `unexpected-failing-tests-fixer` dispatch that did **no measurable work** (no event stream, `—` tokens, `—` cost). Root cause is two independent defects:

1. **`fix-unexpected-failing-tests` is `category: human`** (`process-flow.yaml:2079`), which forces **interactive** mode (`driver.go:1234`: `Headless: opts.Headless && nodeParams["category"] != "human"`). In an **unattended rehearsal** the interactive `claude` TUI sat on the operator's TTY waiting for input that never came → ~2h14m of dead wall-clock. Nothing in the process bounds how long a human-category dispatch may block.
2. **The fixer's prompt only models a behaviour-preserving cycle** ("red = a regression; restore the previously-green behaviour, or update a test coupled to a reshaped surface"). It was dispatched from the **new-behaviour** green-verify (`verify-tests-pass`, verifying the brand-new `shouldRejectOrderWithLineQuantityOf100`), where the behaviour was **never green** — the system simply wasn't finished. With no branch for "implementation incomplete" and no exit-on-uncertainty, the fixer/human edited test sources + DTOs, **broke test compilation**, and the next run found zero discoverable tests → `empty test selection` → `TESTS_INFRA_HALT`.

**End result:** A fix-flow dispatch in an unattended run can no longer silently block for hours — at a human gate with no TTY it yields cleanly to a resumable pending-human state — and the fixer prompt gains a correct frame for the new-behaviour case (a "report incomplete" diagnosis) plus an explicit "if uncertain, halt for a human" exit, so it never guesses its way into a broken build.

## Conclusion (corrected)

The first read — "a runaway agent burned ~2h of compute" — is **wrong**. Evidence-backed conclusion:

- The fixer did **near-zero work**. The `—` tokens/cost are the **fingerprint of an interactive dispatch** (token/cost are parsed from the stream-json `type:"result"` event, which **only headless** emits — `driver.go:1841`). So the 2h14m was an interactive, `category: human` dispatch **blocking on an absent operator** in an unattended rehearsal — pure human-wait, not LLM computation.
- The `empty test selection` / `TESTS_INFRA_HALT` is a **downstream symptom**, not the cause. The fix session broke test **compilation** (Gradle rollback stash present), so zero tests were discoverable. The infra-halt was correct — it just fired late and named the wrong villain.
- There are **two independent defects**: (1) **process/runtime** — nothing bounds a human-category dispatch in an unattended run, so it stalls silently for hours; (2) **agent instructions** — the regression-framed fixer is dispatched into the new-behaviour green-verify where its frame doesn't apply and it has no exit-on-uncertainty, so it guesses and corrupts the build.
- **Could BPMN have prevented it?** It can't make a human appear, but it *can* stop a human-gate blocking forever (run headless / fail-fast when there's no TTY), pre-flight that the named test is discoverable before dispatching anyone, and halt on no-progress/scope-drift. The single highest-value change is **Step 1** — it would have ended this run in minutes regardless of the frame-mismatch.

Notes that informed the steps below:
- **Token/cost `—` = interactive, by design** — not a capture bug. Headless-only (`clauderun.go` result event; `driver.go:1841`).
- **"Not discoverable" = "didn't compile"**, not "didn't exist" — method present at `PlaceOrderNegativeIsolatedTest.java:16`; Gradle `compileTransaction/stash-dir` rollback stashes prove a failed test compile.
- **A wall-clock bound belongs at the dispatch layer, not the prompt** — the model can't see elapsed time. The prompt gets a *behaviour* bound ("one diagnosis, smallest single fix, then exit; halt-for-human if uncertain"). *(Resolved Q3: no time-based watchdog is built — the no-TTY yield removes the stall and Step 4's behaviour-based no-progress/scope-drift guard covers the fix loop; a general wall-clock backstop is a noted follow-up.)*
- **First-run and retries share one classification path** — `verify-tests-pass` re-enters the same `RUN_TESTS → GATE_TESTS_OUTCOME` nodes each loop, and the empty-selection classifier (`verify_classify.go`) is a stateless pure function, so empty selection halts as `infra` identically on the first run and every retry. *(Resolved Q4 + Step 2: the original pre-flight discovery guard was dropped as redundant with this existing classifier — only its halt message is improved.)*
- **A halt here is resumable, not destructive.** The engine has a **git-state-derived resume mechanism** (`internal/atdd/runtime/driver/scoped.go`): every phase commits as it completes, and a resume skips phases already DONE in the committed tree and re-enters at the first unfinished slice. Run #69's committed phases are `ACCEPTANCE TESTS` → `DSL` → `SYSTEM DRIVER ADAPTERS` (`527a9f2c`); the hung fix/verify step is after that and was never committed. So halting at the human gate loses **nothing earned** — a human resumes later, the committed phases are skipped, and they re-enter the human gate *with an operator present*. The only thing discarded is the broken uncommitted fixer edits (which we want gone — they're what broke test compilation). This means "human can proceed" is satisfied by **resume**, not by blocking a live TTY for hours (which an unattended overnight session can't usefully do anyway — there is no mid-dispatch checkpoint).

## Outcomes

- A **human-category dispatch in an unattended run** (rehearsal / CI / any non-TTY context) does not block indefinitely: on no-TTY it **yields cleanly to a pending-human terminal state** (frees the machine, discards uncommitted edits, resumable from the last clean commit) — a normal outcome, never a multi-hour silent stall and never an error-halt.
- The eventual outcome is attributed to the **real** cause (unattended human gate / incomplete implementation), not to a downstream `empty test selection` infra symptom.
- The `unexpected-failing-tests-fixer` prompt is correct for **every** cycle that dispatches it — including new-behaviour green-verify: a third diagnosis ("behaviour never implemented → report incomplete / hand back") covers the case its two original readings missed, so it never "guesses a side."
- The fixer has an explicit **exit-on-uncertainty** path (halt-for-human envelope) so a single pass bails instead of editing blindly and corrupting the build.
- An **empty test selection** still becomes an immediate halt **before** any fixer is dispatched — already delivered by the existing classifier (`infra → TESTS_INFRA_HALT`), now with a clearer operator-actionable message ("named test(s) not discoverable — did they compile / are they named correctly?").

## Open questions — RESOLVED (via /refine-plan 2026-06-16)

All four are decided. Decisions are folded into the Steps above; full rationale retained below for traceability.

1. **Unattended human-category behaviour — what happens when the human isn't at the computer?** ✅ *resolved.*
   **Decision: yield cleanly to a "pending-human" state — do not block, do not error-halt.** Reframing established that "halt" was the wrong word: real human-in-the-loop systems (a CI manual-approval gate, a deploy approval) don't fail and don't burn the executor — the job *pauses*, the runner is **freed**, and the work item sits in a waiting state until a human acts on it. So when a human-category dispatch is reached with no operator TTY, the run ends in a normal, expected **pending-human terminal state** (not a failure like `TESTS_INFRA_HALT`), frees the machine, and is resumable later via the existing git-state resume (`scoped.go`).
   **On resume: re-enter from the last clean commit.** The committed phases are skipped, and the **uncommitted in-progress edits are discarded** so the human re-enters the human gate from a clean, compiling tree — not the half-broken state a timed-out attempt left behind (this is precisely what would have prevented run #69's broken-build cascade). The only thing lost is the broken uncommitted fixer edits, which we *want* gone.
   Rejected: (b) error-halt framing — it's a clean yield, not a failure; (c) run headless — conflicts with the *"never bypassable"* intent the human category encodes; (d) keep waiting — the current 2h14m stall, this is the bug.
   **Detection: no-TTY auto-detect** — check whether stdin is an interactive terminal; non-TTY (rehearsal / CI / cron / piped) yields to pending-human automatically. Zero config, can't be forgotten, fixes #69's exact failure mode (an unattended run that behaved as interactive). No explicit `--unattended` flag.
   **Notification: deferred** — out of scope for this plan. The pending-human state is visible in the run status / summary / exit code; push/email can be added later as its own plan.
2. **Prompt fix vs routing fix (Step 3a vs 3b).** ✅ *resolved → 3a (prompt branch + escape hatch).*
   **Design intent confirmed from the YAML:** `verify-tests-pass` (`process-flow.yaml:1540`) is a single **shared, caller-blind** process. It's called by new-behaviour cycles (`implement-and-verify-system`, narrowed to the new AT's `test-names`), by refactor/structural cycles (full regression), and by the test-layer cascade — and **every `test-outcome == fail` routes to the same `fix-unexpected-failing-tests` → `unexpected-failing-tests-fixer`** (`1594`), regardless of caller. So the fixer is *de facto* the **universal red-responder**; there is no separate routing for "new behaviour still red" vs "refactor regression" today.
   **Decision: 3a.** Keep the fixer as the universal red-responder and make its prompt cover the case it actually receives, plus an exit-on-uncertainty escape hatch. Rejected **3b (routing)** because — given the shared, caller-blind `verify-tests-pass` — rerouting only the new-behaviour case would require teaching `verify-tests-pass` the caller's cycle-kind (e.g. a `fix-process: ${...}` param every caller binds), touching a shared process and every call site; that's a structural change that risks new bugs in flows that currently work. The high-value fix is the **exit-on-uncertainty rule** (it directly stops the #69 guess-and-corrupt), which 3a delivers in one contained instruction file. (3b remains the *philosophically* cleaner "builder finishes features, fixer only repairs regressions" design, but is explicitly **not** pursued here.)
3. **Watchdog scope.** ✅ *resolved → no time-based watchdog in this plan.* The actual #69 stall (a human dispatch blocking forever) is **already removed by Q1** — no-TTY now yields immediately, so a human-specific idle timer is redundant. A *general* per-dispatch wall-clock backstop for all agents would be a brand-new subsystem (there is no `timeout`/`budget`/`deadline`/`no-progress` concept anywhere today — only `max-visits` and the engine-wide `maxDispatchesPerProcess` *count* caps) and a blunt check (a legitimately slow agent looks identical to a stuck one). Instead, rely on **Q1** (removes the stall) + **Step 4** (a smarter *behaviour-based* no-progress/scope-drift guard for the fix loop, which needs no clock — the model can't see wall-clock anyway). A general wall-clock backstop is recorded as a possible **follow-up**, not built here.
4. **First-run infra classification.** ✅ *resolved → confirmed by code.* The classifier `classifyShellErr` (`internal/atdd/process/actions/verify_classify.go:167`) is a **pure function** of `(stderr, err)` with **no first-run/retry state**. Empty selection is matched by two patterns — `verify_classify.go:135` (`no tests? (found|were executed|match…)`) and `:148` (`0 tests executed for the given selection`, the runner's own marker for tools that exit 0 on a zero-match filter) — and returns `classInfra` **identically on every invocation**. So empty-selection buckets as `infra` on the first run and every retry alike; there is no asymmetry to fix. This corroborates the #69 story: the genuine first `fail` (tests ran, one failed) matched no infra pattern → `classRed` → fixer dispatched; only after the fixer broke compilation did a run produce "No tests found" → `classInfra` → `TESTS_INFRA_HALT`.
   **Consequence for Step 2 (flagged):** the gateway already routes `infra` straight to `TESTS_INFRA_HALT` *before* any fixer (`process-flow.yaml:1595`), so an empty selection **already** halts without dispatching a fixer — on any run. This makes the Step 2 pre-flight guard **partly redundant** with the existing classifier; its only marginal add is an *earlier, more precisely-worded* halt ("named test not discoverable — did it compile?") vs. the existing "empty test selection" label. See Step 2 for the scope decision.
