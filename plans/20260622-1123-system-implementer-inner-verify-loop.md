# 2026-06-22 11:23:47 UTC — system-implementer owns an inner red→green loop (no more compile-only "done")

## TL;DR

**Why:** In rehearsal #68, `system-implementer` (api, opus, 6m58s) ran only `gh optivem system compile`, satisfied a *static* completeness checklist, declared "implementation is complete," and exited — shipping code that failed 4/6 acceptance tests at runtime (`Order line for SKU 'DELL-XPS' not found`). The orchestration caught it and routed to a human-gated fixer, so the unattended run paused. Compile-green ≠ acceptance-green, and the agent had no inner feedback loop to discover the gap.
**End result:** `system-implementer` converges its own `${channel}` acceptance slice to green against a *real* test run before declaring done (warm-context debugging it's uniquely positioned to do), using one narrow self-verify command; the orchestration keeps an independent post-implement verdict and a hard iteration bound, and only a genuinely stuck case reaches the human-gated fixer.

## Outcomes

What we get out of this — the goals and deliverables:

- A `system-implementer` dispatch never reports "done" on compile alone; its authoritative done-signal is a passing `acceptance-${channel}` run, not its own judgment.
- The agent self-verifies by calling the **existing** `gh optivem system build` → `system start --restart` → `test run --suite=acceptance-${channel}` commands — the same trio the orchestration runs — so its inner verdict is identical by construction to the orchestration's outer verdict. No new command surface; it does **not** get raw docker/Flyway controls (those commands already wrap `internal/build/runner`).
- The static "name a code path for each assertion" checklist is demoted from gate to heuristic hint; the real gate is the run.
- The orchestration's post-implement build→start→run remains an **independent** confirmation (never trusts the agent's self-report), bounded by a hard cap; the intentional human-approval gate before the `unexpected-failing-tests-fixer` is preserved as the final escalation, reached only after the bound is exhausted.
- Fewer wasted ~7-minute opus dispatches that ship a first guess and burn a full verify cycle; fewer unattended-rehearsal pauses caused by an avoidable compile-only miss.

## Open questions

- **Capability boundary — RESOLVED → agent-owned inner loop (Option A).** First-principles decision: the agent that wrote the code owns the convergence loop (live working-memory debugging against a real test run); an external run owns the verdict + hard bound. Two independent arguments land on A:
  - *Convergence:* debugging is a live-hypothesis activity — the author agent holds the causal model of its own choices and can connect symptom→cause across iterations. Re-dispatch debugs the author's code from a commit message.
  - *Token efficiency:* A loads the expensive context **once** and serves later iterations from the warm prompt cache (~10% rate); BPMN re-dispatch repays the cold-start (re-reads AT/DSL/driver/production files fresh) on **every** iteration. Tie at 0 extra iterations; A wins from the 1st extra iteration and the gap widens. (Per-#68 dispatch: 80.3k fresh / 2423.6k cached / 24.5k out — the 2.4M is amortized in A, repaid per-iteration in B/C.)
  - Consequence to handle in execution: this makes the implementer a *system-running* agent, so the deliberate **"compile, don't run" convention in `internal/atdd/assets/runtime/shared/preamble.md:31-39` must be rewritten for the implementer** — not silently violated. (Rejected: B = BPMN cold re-dispatch = "guess again from scratch"; C = BPMN warm re-dispatch feeding diff+failures forward — only attractive if we keep the compile-only boundary, which we are not. Re-dispatch's one edge is trace visibility per iteration, independent of tokens.)
- **Verify-command shape — RESOLVED → B (reuse existing commands).** No new command. The agent calls `system build` → `system start --restart` → `test run --suite=acceptance-${channel}` (same trio as the orchestration → verdict parity, no drift). A thin `verify` wrapper can be added later *if* the sequence proves error-prone in practice — not built speculatively now.
- **Iteration bound — RESOLVED (captured in Step 3):** budget-primary + "stop when not progressing" (N consecutive runs with no newly-passing test). Hard backstop = **reuse `clauderun`'s standard dispatch ceiling** (no bespoke per-agent ceiling — the "stop when not progressing" rule already kills wasted loops early, so a special case buys little).
- **Cost posture — RESOLVED.** A self-verify loop lengthens a single dispatch, but A is *cheaper* than the re-dispatch alternatives once >1 iteration is needed (context loaded once + warm-cache amortization vs. cold-start repaid per re-dispatch — see capability-boundary note). Net: fewer wasted full verify cycles and fewer avoidable pauses, at lower total tokens. Accepted.
- **Model-tier scope — RESOLVED:** the verify loop applies to **every** channel/tier, not just the `common: true` first channel. A later (sonnet, `common: false`) channel can still ship an adapter delta that compiles but fails its slice — same failure class. The loop simply targets that channel's `acceptance-${channel}` slice; no tier-specific logic needed, and the warm-cache amortization holds regardless of tier.
