# 2026-06-19 11:39:00 UTC — Channel-aware system unroll: derive implement/verify channels from the tests, not static config

🤖 **Picked up by agent** — `ValentinaLaptop` at `2026-06-19T13:45:37Z`

## TL;DR

**Why:** The ATDD orchestration unrolls one implement-and-verify-system dispatch per *configured* channel (`channels: api, ui`), blind to which channels each ticket's acceptance tests actually registered for. When a ticket's tests are channel-specific (e.g. API-only negative/error-message tests), the orchestration still dispatches a system-implementer for the absent channel **and** verifies that channel against a suite that cannot contain the test — wasting compute and then hard-halting on `TESTS_INFRA_HALT`.
**End result:** Each per-channel implement-and-verify clone **self-guards inside the cycle** (mirroring the existing external-systems pattern): a `resolve-channel` service-task stamps `channel-touched`, and a `GATE_CHANNEL_TOUCHED` gateway no-ops a channel that none of the ticket's acceptance tests register for — straight to a skip end-event. An API-only ticket therefore does no UI implement and no UI verify (no wasted compute, no `TESTS_INFRA_HALT`). An upfront fail-loud guard rejects a test that targets an unconfigured channel, and a command-level safety net turns any residual `--test`/channel mismatch into a clean skip instead of a crash.

## Outcomes

What we get out of this — the goals and deliverables:

- The channel unroll stays **static** (one clone per configured channel), but each clone gains an **in-cycle membership guard** — `resolve-channel` + `GATE_CHANNEL_TOUCHED` — so an untouched channel no-ops to a skip end-event. This mirrors the proven `UnrollExternalSystems` design (`resolve-external-system` + `GATE_EXTERNAL_SYSTEM_TOUCHED`) rather than inventing a new seam.
- A channel is **"touched" iff `acceptance-<ch>` contains ≥1 of the ticket's relevant acceptance tests** — a membership query on the *compiled* tests, reusing the `testselect`/runner suite-resolution machinery (language-agnostic; no per-stack `forChannels(...)` text parsing).
- An **API-only ticket does no UI work at all**: the UI clone hits the gate and skips — no UI `system-implementer` dispatch (no wasted ~12 min / ~$1.50 like rehearsal #76), no UI verify, no `TESTS_INFRA_HALT`.
- A **UI+API ticket still runs both** channels — no regression for multi-channel tickets.
- The **API-only convention for negative/error-message acceptance tests keeps working untouched** — the test-writer is *not* forced to register tests for channels they don't belong to.
- **Both per-channel unroll sites are covered:** the system cycle (`UnrollSystemChannels`, `channels.go:68`) and the test-side driver-adapter cycle (`UnrollSystemDriverAdapterChannels`, `channels.go:125`), which shares the same blind-to-registration assumption.
- **Fail-loud upfront guard** (the channel analog of `validate-external-systems-registered`): if a ticket's acceptance test registers for a channel **not** in configured `channels:`, the run hard-errors (no-silent-skip) rather than letting that channel go unverified.
- **Safety net (defense-in-depth):** `gh optivem test run --test=X --suite=<s>` exits cleanly (skip) when `X` matches zero tests in `<s>` **but exists in another channel/suite**, instead of exit 1 / infra-halt. A genuinely non-existent / misspelled test (exists nowhere) still **fails loud**. Also protects manual/direct CLI invocations the orchestration gate doesn't cover.
- A regression test reproducing the #76 shape (API-only new test + `channels: api, ui`) that goes green with the fix.

## Resolved decisions

1. **Discovery mechanism** → a **channel-membership query on the compiled tests** (`acceptance-<ch>` contains ≥1 of the ticket's tests), reusing `testselect` suite-resolution + the runner's listing. *Not* static `forChannels(...)` text parsing — that is language-specific and brittle across the Java/.NET/TS stacks; the runner already knows channel membership at runtime.
2. **Where the derivation runs** → **inside each unrolled clone**, mirroring `UnrollExternalSystems`: keep the static unroll, add `resolve-channel` + `GATE_CHANNEL_TOUCHED`. Chosen over feeding a derived list into the load-time unroll because the unroll runs before test metadata exists and a preceding gateway cannot read baked call-activity params (`channels.go:165` note). The self-guard is the engine's own established pattern for "partially-touched unrolled set."
3. **Empty / unconfigured-channel behavior** → **fail loud**, per the repo rule. Mirror `validateExternalSystemsRegistered`: an upfront guard hard-errors if a ticket's test registers for a channel absent from configured `channels:` (no-silent-skip). The touched-gate handles the benign direction (configured channel with no matching test → skip); the guard handles the dangerous direction (test for an unconfigured channel → error, never silently unverified).
4. **Core + net sequencing** → **safety net (Step 2) ships first**, independently — smallest, lowest-risk, and alone converts #76's halt into a skip. The in-cycle guard (Steps 3–4) lands after and is the architecturally-complete fix (it also removes the wasted implementer). **Both are kept**: with the guard, the orchestration never verifies an untouched channel; the net remains as defense-in-depth and to protect manual/direct `gh optivem test run` invocations.
5. **Other unroll sites** → **in scope.** `UnrollSystemDriverAdapterChannels` (`channels.go:125`) shares the same blind-to-registration assumption and gets the same `resolve-channel` + `GATE_CHANNEL_TOUCHED` guard (Step 4). Its "touched" semantics may additionally key on whether the channel's acceptance tests touched a system driver port — reconciled against the Step-1 primitive.
