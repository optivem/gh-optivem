# Go CLI — retry transient `gh` errors — 2026-04-21 13:10 UTC

## Motivation

`gh-optivem` is a Go CLI that shells out to `gh` from `internal/shell/github.go`
(and `internal/steps/*.go`). Every `gh api`, `gh repo`, `gh run` call is
exposed to the same transient 5xx/network failures that motivated the bash
`gh_retry` wrapper in `optivem/actions` (see its `shared/gh-retry.sh`).

The bash workflow layer in this repo (`.github/workflows/gh-*-stage.yml`,
`scripts/cleanup-orphans.sh`) was ported to `gh_retry` on 2026-04-21. The Go
CLI was deliberately deferred — it's a different problem class that needs Go
code + tests, not a bash wrapper.

Rate-limit **pre-check** is already implemented as `CheckRateLimit()` in
`internal/shell/github.go:112`. What's missing is the **transient-error retry**.

## Scope

- `internal/shell/run.go` (or equivalent) — add a `RunWithRetry` helper that
  wraps `Run` / `RunCapture` and retries transient failures with the same
  policy as bash `gh_retry`:
  - Up to 4 attempts, 5s → 15s → 45s backoff.
  - Retryable: HTTP 5xx, timeouts, connection resets, TLS handshake, DNS
    transients.
  - Hard-fail (no retry): HTTP 4xx including 403 rate-limit — pass through so
    `CheckRateLimit` can handle the next iteration.
- Route every `gh api|release|workflow|run|repo|pr|issue` call through the new
  helper. Keep `gh api rate_limit` raw (used by `CheckRateLimit` itself).
- Unit tests: mock `exec.Command` results covering (a) immediate success,
  (b) transient failure then success, (c) transient failure exhausting retries,
  (d) hard-fail pass-through.

## Non-goals

- No change to `gh extension`, `gh auth`, or pure-git invocations.
- No change to the bash layer already ported on 2026-04-21.

## Open decisions

- Where to put the helper: extend `internal/shell/run.go` vs new
  `internal/shell/ghretry.go`. Recommended: new file — keeps the policy
  separate from the generic `Run` so non-`gh` commands aren't accidentally
  retried.
- Should `CheckRateLimit` itself use the retry wrapper? No — it probes
  `gh api rate_limit`, which is the one call that must stay raw to avoid
  recursion and to surface transient probe failures immediately.
